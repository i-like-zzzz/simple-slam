# simple_slam 开发说明

这份说明只记录当前这套前端实现为什么这么拆，不写空泛口号。

## 当前目录怎么分

- `include/simple_slam/frontend`
  放局部前端。现在已经有 scan 预处理、里程计预测、相关匹配和关键帧判定。
- `include/simple_slam/mapping`
  放子图和地图表达。当前 `Submap2D` 用的是固定大小局部占据栅格。
- `include/simple_slam/optimization`
  先保留位姿图数据容器，后面后端做起来时这里会继续长。
- `include/simple_slam/backend`
  先放后端接口，不急着塞实现。这样前端和后端边界先固定住。
- `include/simple_slam/system`
  放系统级别的模式和调度相关定义。现在已经把 `mapping` / `localization` 模式抽出来了。

## 当前前端做了什么

现在的前端链路是：

1. 接 `/scan`
2. 过滤无效量测并做简单体素降采样
3. 用 `/odom` 做位姿预测
4. 在当前活动子图上做实时相关匹配
5. 按位移和角度阈值决定是否生成关键帧
6. 关键帧插入活动子图，并在话题上发布关键帧 Marker

这版的目标不是“参数堆满”，而是先把链路跑顺，后面再逐步把精细部件替换掉。

## 为什么先做关键帧 Marker

前端调试最怕“看不到自己在什么时候插帧”。现在关键帧会发到：

- `trajectory`
- `keyframes`

其中 `keyframes` 里最新关键帧会用更亮的颜色显示，方便在 RViz 里盯前端行为。

## 运行模式

- `mapping`
  正常建图，前端会更新活动子图。
- `localization`
  先把接口留好。目前只关闭地图更新，后面会接“加载已有地图 + 仅定位”。

## 这一阶段刻意没做的东西

- 没上 Ceres 精配准
- 没做回环检测
- 没做后端优化线程
- 没做地图持久化

这些都不是忘了，而是要等前端的输入输出边界稳定之后再接，不然代码会很快变乱。
